# Decision D11: Removing TimesFM from the Ensemble

The original project specification envisioned a three-model ensemble combining an LSTM, a TCN-based MultiHorizon forecaster, and a LoRA-finetuned TimesFM foundation model.

## Decision
TimesFM is dropped from the active ensemble. The final system is a Two-Model Ensemble (LSTM + MultiHorizon TCN) with equal weights.

## Rationale
1. **Incomplete Implementation:** The `lora_finetune_timesfm()` function in `timesfm_model.py` is an unwritten stub. Including an untrained, generic foundational time-series model (which knows nothing of solar physics or X-ray flux profiles) as 30% of the ensemble weight would actively degrade prediction performance.
2. **Hardware Constraints:** True LoRA fine-tuning of the 200M parameter TimesFM model requires a CUDA-capable GPU with ≥8GB VRAM. This cannot be guaranteed on the target laptop hardware ("Nitro 5"), risking OOM crashes during the training pipeline.
3. **Dead Branch Bug:** The previous `ensemble.py` implementation had a fatal logic bug in its fallback handling: even if a valid TimesFM model were passed, the ensemble code checked for `.predict_proba()` (which TimesFM lacks, as it uses `.forecast()`), causing it to silently fall through to a `[0.25, 0.25, 0.25, 0.25]` uniform probability fallback.

## Implementation
- `src/forecasting/ensemble.py` is updated to accept `timesfm_model=None`.
- The dead code branch checking for `timesfm_model` has been replaced with a 2-model weighting scheme (50% LSTM, 50% MultiHorizon).
- The ensemble training script (`scripts/train_ensemble.py`) instantiates and evaluates only the 2-model variant.
